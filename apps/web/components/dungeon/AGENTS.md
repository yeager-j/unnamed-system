# `components/dungeon` — dungeon run console + player watch

**Dungeon run console + player watch** (UNN-463/464/467; spatial M2 exploration).

Organized **by phase/viewer**: root holds the three route-facing entries (run-console = DM console rendering the exploration Play phase, prep = DM draft placement, watch = player read-only view) + explore-sheet-column (the watch's own-sheet column during exploration — UNN-566: the shared `OwnedSheetTabs` + `OwnerSheetHeader` over the Explore **cards**, stacked; the `ExploreTab` root is not reused because its viewport-keyed card grid overflows a narrow rail). The watch is **one surface for both phases** (UNN-604): its phase is read from `DungeonSnapshot.combat` (the live-fight linkage, UNN-603), and during a fight the body swaps to `combat/watch.tsx`'s `DungeonCombatWatchBody` — whose column is the encounter's `CombatSheetColumn` — with a guarded `router.refresh()` used only to rehydrate the other phase's server props, never as a fork. shell/ is the persistent DungeonConsoleShell + DungeonSidebarSlot (UNN-488, eases --sidebar-width across phase swaps).

explore/ is the Play phase (body, party-sidebar, zone-sheet, exit-row, use-dungeon-console, dispatch-event). combat/ is the **combat phase on engine v2** (UNN-536): body (drives the shared `combat/console/use-combat-console`, injecting the three-row `endDungeonCombatAction` as the one route-varying seam), sidebar (the shared CombatantRail), end-combat-dialog, encounter-staging (the **client-side** pre-combat staging at `app/dungeon/[shortId]/encounter/` — the shared `EnemyCatalogPanel` bestiary over a dungeon-id-keyed queue whose groups carry a zone (`hooks/use-dungeon-enemy-queue`), no persisted draft; "Start an encounter" on the Play bar navigates there → `startDungeonEncounterAction` mints an already-live encounter — UNN-541), and watch (`DungeonCombatWatchBody` — the delve combat player view as the **C3 composition**, UNN-604: the fog map stays the battlefield; the board is the parent's dungeon snapshot, the pieces are its own fogged-endpoint `useEncounterSnapshot` subscription joined by `zoneId`, and it drives the parent's board `refetch` when the shared Instance advances since combat writes ping the encounter channel). The **DM console's** combat-vs-explore fork is decided once at its route loader (the DM's own actions re-run it); the **watch** has no server fork — its phase is client-observable snapshot state.

canvas/ is the React Flow layer: shared core at its root (canvas, build-nodes, types, zone-card-frame, connection-edge, floating-edge-handles, token-glyph, viewport-store, mode-toggle, edit-canvas) + a subfolder per surface — explore/, combat/ (UNN-536: zone-node/token-chip/context/turn-bar/spine-panel — the live battlefield, tokens grouped from the console RosterView by occupancy zone; `DungeonCanvasMode` is now `play | combat`), and watch/ (the player view, formerly "fog"; the fog-of-war *mechanic* keeps the engine's ConnectionFogState name; also owns engaged-cluster + combat-zone-node — its canvas takes a `DungeonWatchCanvasMode` of `explore | combat`, the combat arm rendering the redacted `WatchCombatant` pieces on the same board). Each surface holds its own zone-node/token-chip/context.

Imports cross-reference via the @/components/dungeon/… alias. Rendered by app/dungeon/[shortId]/ (DM) + app/c/dungeon/[shortId]/ (watch).
