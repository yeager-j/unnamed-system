# `components/combat` — shared combat UI kit

**Shared combat UI kit** (UNN-492): the route-agnostic combat components rendered by BOTH the mapless encounter (`components/encounter/`) and the spatial dungeon combat canvas (`components/dungeon/`) — it depends on neither experience.

**Organized by concern** (folder names drop the redundant combatant-/combat- prefixes): console/ (the **headless** core — use-combat-console + dispatch-event + pc-ping), drawer/ (combatant-drawer + the actions/conditions/counters/engagement/position/vitals sections), rail/ (combatant-rail + row), conditions/ (controls = the shared ailment/battle-condition editor + state-display = its read-only peer), dialogs/ (start-combat / end-combat / end-of-turn-modal), setup/ (import-pcs-panel + panels stub), watch/ (the player-watch shared parts — owned-sheet-tabs mounts one owner-mode `EntityWriteProvider` per owned character and is rendered by **both** watches, owner-sheet-header = the identity + HP/SP + adjust-pool masthead, combat-sheet-column = both watches' combat own-sheet body, player-turn-order + watch-enemies-rail = the battlefield's turn tracker + redacted enemy roster rendered by both the mapless watch and the delve combat watch (moved from `encounter/` in UNN-604), + the owned-sheet-refresh hook that re-pulls the server-resolved sheets when the polled snapshot implies a different sheet — its pure key is `domain/combat/view/owned-sheet-refresh-key.ts`, with the rest of the snapshot folds. That key covers zone effects **and** the owned combatant's pools: with no `ABLY_API_KEY` the `character:{shortId}` ping never arrives, so the hook is the sole path reconciling a DM's damage into the column), controls/ (engagement, side-toggle, zone-enchantment), and enemies/. turn-order-strip stays at the root (exports ConsolePhase).

The watch columns compose the **v2 sheet components** directly (`SkillCastSection`, `MechanicWidget`, `VitalsBlock`, the Explore cards) — never a tab root. A tab root owns page chrome (its own padding, a sticky bar, viewport-keyed `lg:` card grids) that overflows a 340px rail; extract the shared body instead.

The enemies/ bestiary (UNN-346) is a three-column master-detail whose EnemyCatalogPanel is the shared inner surface: it owns **browse** only (searchable/family-filtered/level-grouped list + statblock card) and takes the third column as a `rail` node, so neither the queue's identity nor what "commit" means reaches it (UNN-541). EnemyQueueRail is the shared cart — **id-keyed** groups with `qualifier`/`detail` per row plus `headerAccessory` / `children` / `commitLabel` slots. Two consumers compose them over the shared localStorage store (hooks/use-staged-enemy-queue + the pure ops in hooks/staged-enemy-queue): the mapless browser wrapper in encounter/ (groups by enemy key, keyed by encounterId) and the delve's staging surface in dungeon/ (groups by enemy × zone, keyed by dungeonId). Browse reads through the app-side seam (UNN-583): `domain/combat/view/enemy-catalog-view.ts` re-exports the row builder + family vocab and folds the selected key into a statblock (`selectedEnemyStatblock`), so the panel imports only `@/lib/**`. `enemy-statblock.tsx` (`EnemyStatblock`) renders an `EnemyStatblockView` (`domain/combat/view/enemy-statblock-view.ts`, projected off `resolveEntity(getEnemy(key))`); its Skills are pre-folded to `SkillCardView`s (via `domain/combat/view/skill-card-view.ts`) and rendered by the shared `ResolvedSkillRow`.

**combat conditions (ailments + battle conditions) are DM-only** — conditions/controls is the shared presentational editor; the DM drawer's conditions-section is its sole editing caller (UNN-467), the watch renders it read-only.
