**Mapless-encounter feature** (UNN-492, split out of `combat/`): the unique shell for the no-map encounter, rendering the shared `combat/` kit.

The DM console (UNN-335; on engine v2 since UNN-535) ‚Äî encounter-setup draft shell (v2 wire gestures via use-encounter-setup + combatant-setup-row; add-then-place through the `placeCombatant` upsert, Start gated by the engine's `isRosterFullyPlaced`), combat-console (live, driven by the shared combat/console/use-combat-console over one optimistic `EncounterState`), and ended-stub ‚Äî rendered by app/campaigns/[campaignShortId]/encounter/[shortId]/ (its co-located encounter-access.ts holds the DM-only `getEncounterForDM` loader ‚Äî dissolved v2 Session + instance + `participantMeta`, 404-collapsing on campaign-pairing ‚Äî the console + setup sub-route memoize; enemy-catalog-browser is the mapless wrapper around the shared EnemyCatalogPanel, committing via `addCatalogEnemiesAction`, route .../encounter/[shortId]/setup/).

The **player watch view** (UNN-322/323/324) ‚Äî encounter-watch (client root + draft/live/ended status branch, consumes domain/combat/use-encounter-snapshot) ‚Äî rendered by app/campaigns/[campaignShortId]/encounter/[shortId]/watch/ as a **two-pane** surface: the signed-in viewer's own combatant sheet(s) on the left (the shared combat/watch/ `CombatSheetColumn` ‚Äî tabs when they own >1 combatant here, each an owner-mode `EntityWriteProvider` over the **same** v2 sheet components /characters/[shortId] renders) and the battlefield on the right (the mapless zone-layout grid here + the shared combat/watch/ `PlayerTurnOrder` and `WatchEnemiesRail` ‚Äî both moved to the kit in UNN-604 since the delve combat watch renders them too ‚Äî all shaped by domain/combat/view/watch-layout.ts's `buildWatchView`). The left column simply isn't rendered for a spectator/signed-out viewer (battlefield full-width).

All combatant data is the **redacted** v2 SpatialEncounterSnapshot (per-relationship component drops decided by the engine's visibility table in lib/db/queries/load-encounter-snapshot.ts's `getEncounterSnapshot`; dropped keys are structurally absent), polled/pinged from the public app/api/encounter/[shortId]/snapshot route with a composite version (`encounter √ó instance √ó durable vitalsVersions`) the client equality-compares; the viewer's own sheets come from the same module's `loadOwnedEncounterSheets` (UNN-566), each resolved with its combatant's party composition + Zone-Enchantment effects and carrying that `ResolveContext` to the client so the provider's optimistic re-fold matches the DM drawer's numbers. On the watch the player edits **vitals/SP/mechanic** (through the sheet's own owner controls, `useEntityWrite` descriptors ‚Üí the entity door); combat conditions render read-only.
