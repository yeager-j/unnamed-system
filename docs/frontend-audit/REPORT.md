## Executive summary

**The architecture is healthy; the line-level findings cluster into a few systemic gaps.** All five subsystem reviews came back positive — the optimistic write machinery, realtime/invalidation stack, loading/hydration spine, combat state model, and engine seam each solve their hard problem once, at a choke point, and the DX walkthrough found the paved path for a new field is real. The architecture findings are P2/P3 refinements, with one P1: the `lib/actions` layer's per-field boilerplate plus a README that has drifted from reality.

**The P0 story is dominated by one bug class.** Five of the 13 verified P0s are recurrences of the documented UNN-226 class — handlers composing full post-state from optimistic values (ailment editor, enemy vitals, builder virtues, combat-console version forwarding) — the rule exists in CLAUDE.md but didn't stamp out the pattern. A second systemic gap compounds the rest: there are **no route error boundaries anywhere under `apps/web/app`**, and several write paths don't catch rejected Server Actions, so a network failure can blank a page instead of showing the retry toast.

**Accessibility is the weakest dimension** (64 findings, most of the P1 themes): recurring patterns rather than one-offs — `aria-label` misuse that hides real content, zero live regions on a polling-driven app, keyboard-unreachable tooltips and hidden file inputs, and contrast-failing stacked opacities.

**Debt findings are numerous but shallow** — convention sweeps (labels outside `labels.ts`, inline data shaping that belongs in engine helpers, dead exports, near-duplicate components) that fit naturally into a handful of batch tickets.

## P0 — verified user-visible bugs (13)

### `apps/web/components/character-sheet/level-up-dialog.tsx:139-148`

**"Spend them on the Archetypes tab" link cannot switch the tab — sheet tabs are client state, not routing**

The dialog renders `<Link href="?tab=archetypes" ... onClick={() => onOpenChange(false)}>Spend them on the Archetypes tab</Link>`. But the active sheet tab is in-memory client state: `SheetNavProvider` (components/character-sheet/sheet-nav-context.tsx:31) holds it as `useState<SheetTabKey>(defaultTab)` and is rendered un-keyed by app/c/[shortId]/page.tsx:117 (`<SheetNavProvider defaultTab={resolveTab(tab)}>`). A search-param-only soft navigation re-renders the page RSC with the new `defaultTab`, but React preserves the provider's `useState`, so `activeTab` never changes — `defaultTab` is only read on mount. The provider's own JSDoc states the constraint this Link violates: "The four tabs are in-memory client state — not routing — so a navigation command can't `router.push(?tab=)` to switch them; it calls setActiveTab here instead" (sheet-nav-context.tsx:12-14), and the command-palette registry correctly uses `ctx.setActiveTab(tab)` (lib/commands/navigation.ts:45). Net effect: after leveling up, clicking the CTA closes the dialog, scrolls to top, and rewrites the URL to `?tab=archetypes` while the Combat tab stays visible — the link silently no-ops AND leaves the URL disagreeing with the rendered tab (useTabUrlSync won't correct it because `activeTab` didn't change, so its effect doesn't re-run). Note the superficially similar link in archetypes/atlas/lineage-atlas.tsx:76 is fine — it navigates from a different route, so the sheet mounts fresh and `defaultTab` applies; only this in-page link hits the bug.

**Fix:** Replace the `Link` with a button (or keep the anchor but intercept) that calls `useSheetNav().setActiveTab("archetypes")` plus `onOpenChange(false)` — the dialog is rendered inside `SheetNavProvider` (SheetHeader → HeaderOwnerActions → LevelUpDialog), so the context is available; `useTabUrlSync` will then mirror the URL automatically. Alternatively key `SheetNavProvider` by the resolved tab in page.tsx, but that re-creates all tab state on every ?tab= navigation — the setActiveTab route matches the existing command-palette pattern.

### `apps/web/components/character-sheet/combat-state/ailment-editor.tsx:66-155`

**Ailment editor composes the full post-state array from optimistic state (UNN-226 class) — concurrent same-field writes are silently clobbered**

Every sub-control in OwnerAilmentEditor builds the complete ailments array client-side from the optimistic value and POSTs it whole: `function dispatch(next: string[]) { write({ edit: { kind: "ailments", ailments: next }, ... action: (expectedVersion) => setAilmentsAction({ characterId, ailments: next, expectedVersion }) }) }`, with the Downed toggle calling `dispatch(withDownedToggled(optimisticAilments, next))` and each row calling `dispatch(withNonDownedSelection(optimisticAilments, ...))`. Downed and the non-Downed selection are two independently-controlled sub-fields (the file's own helpers `withDownedToggled`/`withNonDownedSelection` encode exactly that), yet the write is "client builds the full object" — the shape CLAUDE.md's "Owner-mode writes" rule forbids, and the only combat-state control still using it (FlagRow/BattleConditionAxis/ExhaustionStepper all send per-field intent). The in-tab race is blocked by the shared `pending` disabling all rows, but the cross-tab/cross-device race is live and SILENT: (1) with realtime configured, an invalidation ping forwards the version refs immediately (`mergePingedVersions` in apps/web/hooks/character-version-sync.ts sets `ref.current = version` before `router.refresh()` delivers the new character data), so a click in the ping→refresh window composes the array from the stale base but dispatches with a fresh version token — the write succeeds first-try with no stale signal and overwrites the other tab's ailment; (2) without realtime, `dispatchCharacterWriteWithRetry` handles the resulting "stale" by refetching only the VERSION and re-sending the identical stale-composed payload, converting the optimistic-concurrency conflict into last-write-wins on the array. Either way the other client's ailment change (e.g. it set ["poisoned"], this tab toggles Downed → final state ["downed"]) is dropped with no error, no toast, and an optimistic UI that briefly lies. Multi-tab is an explicitly supported scenario here (UNN-203 BroadcastChannel + UNN-372 Ably exist for it).

**Fix:** Split the write into per-sub-field intent actions merged server-side, mirroring setBattleConditionFlagAction: e.g. setDownedAction({ characterId, downed, expectedVersion }) and setActiveAilmentAction({ characterId, ailmentKey: string | null, expectedVersion }) — the server reads the row's current ailments and applies withDownedToggled/withNonDownedSelection there. "Clear ailments" can remain a whole-field wipe (it is genuinely whole-field intent).

### `apps/web/components/combat/use-combat-console.ts:97-103`

**Forwarding versionRef from a realtime ping before the refresh lands defeats the version guard for absolute-payload events — silent lost update (UNN-226 class)**

`onPing: (data) => { ... if (ping.version <= versionRef.current) return; versionRef.current = ping.version; scheduleRefresh() }` bumps the write token to the remote writer's version immediately, but the new session only arrives when the scheduled `router.refresh()` round-trip completes (typically 100-500ms later). In that window the console still renders the stale session, and several controls compose absolute payloads from rendered state: combatant-vitals-section.tsx:202-208 `onDecrement={(amount) => onAdjust(id, "currentHP", hp.current - amount)}` (adjustEnemyVitals is an absolute set — see packages/game engine reduce/enemy-vitals.ts) and engagement-control.tsx:42-51 `toggle()` which rebuilds the full `targetCombatantIds` list from the rendered `targets`. A DM tap in the window POSTs `expectedVersion = <pinged fresh version>`, so the server guard passes and the stale-derived absolute overwrites the remote write with no rejection, no toast — e.g. tab B sets the ogre to 5 HP, tab A's ping bumps its token, tab A taps "-3" off the still-rendered 12 and persists 9, silently undoing tab B. Multi-writer is an explicitly supported scenario (this hook's own JSDoc: "another writer's change (a player's self-heal, a second DM tab's event) refreshes the page"). The version guard exists precisely to catch this, and the eager ref-forwarding switches it off for the ping-to-refresh window.

**Fix:** Don't write the pinged version into the dispatch token directly; keep a separate `lastPingedVersion` for the dedupe/refresh compare and only advance `versionRef` when the refreshed props actually land (the existing prop-sync effect already does that). A dispatch issued against the older token then gets an honest 'stale' rejection (or, with the finding-1 retry, a correct re-reduce against the fresh row). Alternatively, convert adjustEnemyVitals/engagement to delta/add-remove events so stale-frame composition can't clobber a concurrent absolute.

### `apps/web/components/builder/movements/ortus/virtues-control.tsx:67-101`

**Render-time prop adoption can clobber the local virtue draft mid-burst; next click composes the full 4-virtue payload from the clobbered draft, permanently dropping an earlier click (UNN-226 class)**

```ts
const [draft, setDraft] = useState<VirtueAllocation>(allocation)
const [previousAllocation, setPreviousAllocation] = useState(allocation)
if (allocation !== previousAllocation) {
  setPreviousAllocation(allocation)
  setDraft(allocation)
}
...
function setRank(key: VirtueKey, rank: 0 | 1 | 2) {
  if (draft[key] === rank) return
  applyAllocation({ ...draft, [key]: rank })
}
```

The designed interaction is exactly a three-click burst (+2, then +1, then +1). Each `applyAllocation` POSTs the FULL four-field allocation built from local `draft`. Every successful write calls `revalidateCharacter`, which re-renders the builder layout; when click 1's revalidated RSC payload lands between clicks (one server roundtrip, ~100-600ms — the same cadence as the clicks), the four virtue props change → `allocation` gets a new identity → the adoption block resets `draft` to click-1 state, silently discarding click 2 from the draft. Click 3 then composes `{...draft}` from that clobbered state and writes a full payload that omits click 2's rank; because the payload is the whole object and the dispatch pipeline silently retries it at the fresh version, click 3's write wins over click 2's already-committed write. Net: the player's middle +1 is reverted server-side. The comment on lines 58-66 explains why `useOptimistic` was avoided for exactly this drop-the-intermediate-intent failure, but the adoption-on-prop-echo reintroduces the same hole through a different window. CLAUDE.md's "Owner-mode writes" rule names this exact class: multiple controls (the four virtue rows) composing the full post-state client-side instead of per-field server merges.

**Fix:** Follow the documented combat-state.ts pattern: expose a per-field action (`setVirtueRankAction(characterId, virtueKey, rank, expectedVersion)`) and let the server read the row and merge, so each click carries only its own delta and a prop echo can no longer drop a sibling click from the payload. The UI's cap-gating can keep using the local draft for disabled states.

### `apps/web/components/my-characters/create-character-button.tsx:26-35`

**Rejected server-action promise bypasses the component's own retry toast (no rejection handling in the transition)**

```ts
startTransition(async () => {
  const result = await startCharacterDraftAction()
  if (!result.ok) {
    toast.error("Couldn't start a new character. Try again.")
    return
  }
  router.push(...)
})
```

The toast exists precisely for transient failure ("Try again"), but it only fires when the action *resolves* with a not-ok Result. The most common transient failure — the POST to the action failing (offline, flaky network, deploy in progress) — rejects the promise instead, and there is no try/catch, so the error escapes the async transition and surfaces through React 19's uncaught-transition-error path (the route error boundary) rather than the in-place retryable toast. The same unguarded `await <action>` shape appears in both delete-dialog handlers (covered separately).

**Fix:** Wrap the awaited action in try/catch inside the transition and route the catch to the same `toast.error("Couldn't start a new character. Try again.")` path, keeping the user on the page with the button re-enabled.

### `apps/web/components/shared/adjust-pool-controls.tsx:71-90`

**Enter key in AdjustPoolForm silently fires the destructive (decrement) action, contradicting the visual primary button**

`onKeyDown={(event) => { if (event.key === "Enter") { event.preventDefault(); submit(onDecrement) } }}` — Enter in the autofocused amount input always invokes the decrement handler. Every caller binds decrement to the harmful direction: `header-owner-actions.tsx` and `combatant-vitals-section.tsx` pass decrementLabel="Take damage" / "Spend SP", and `currency-control.tsx` passes decrementLabel="Spend" (wallet). Meanwhile the button grid renders decrement as variant="destructive" on the left and increment as the default/primary-styled Button on the right — so the visually primary action is increment, but the keyboard default is the destructive one. **Fix:** Either remove the Enter binding entirely (require an explicit button click for a two-direction form), or make the Enter-bound action match the visually primary button (e.g. wrap in a `<form>` whose type="submit" button is the one Enter triggers and style it as the default). If Enter→damage is a deliberate combat-speed choice, gate it behind an explicit prop (e.g. enterAction="decrement") so the currency caller can opt out, and document it.

### `apps/web/hooks/use-debounced-auto-save.ts:179-202`

**Failure rollback unconditionally clobbers the current draft, wiping keystrokes typed after the failed save dispatched and desyncing input from server**

```ts
        if (result.ok) {
          lastSavedRef.current = result.value.value
          return
        }
        
        setLocalValue(lastSavedRef.current)
```

(and the same `setLocalValue(lastSavedRef.current)` in the catch block, line 200). The rollback never checks that the draft still equals the `next` value whose save failed. Sequence: user types "A" → debounce fires, save(A) dispatched (or queued behind an in-flight sibling save — the shared per-class queue makes this window longer than one round-trip); user keeps typing, draft = "AB" with a new debounce armed; save(A) fails → `setLocalValue(lastSavedRef.current)` snaps the focused input back to the pre-"A" value, visibly deleting "AB" mid-typing. The armed debounce then fires `performSave("AB")` — which can succeed and set `lastSavedRef = "AB"`, but the success path never restores the draft, so the input shows the old value while the server holds "AB" (prop-sync is suppressed while focused). Worse, with the default `isEmpty: () => false`, a subsequent blur flushes the rollback-injected stale value, overwriting the user's persisted "AB".

**Fix:** Roll back only when the draft hasn't moved past the failed value: capture `next` and use a functional update, e.g. `setLocalValue((current) => (isEqual(current, next) ? lastSavedRef.current : current))` — or compare against a `latestDraftRef` before reverting. Keep the toast either way.

The OG route loads the raw row and renders it with no status check: `const character = await loadCharacterRowByShortId(shortId); const element = character?.portraitUrl ? (<img src={character.portraitUrl} .../>) : (<FallbackCard name={character?.name ?? null} />)`. ### `apps/web/components/combat/combatant-vitals-section.tsx:196-226`

**Enemy HP/Max writes compose an absolute post-state from the optimistic session — the documented UNN-226 stale-closure class**

`onDecrement={(amount) => onAdjust(id, "currentHP", hp.current - amount)}` / `onIncrement={(amount) => onAdjust(id, "currentHP", Math.min(hp.max, hp.current + amount))}` (lines 202-206), and for max HP `onAdjust(id, "maxHP", Math.max(0, hp.max - amount))` / `onAdjust(id, "maxHP", hp.max + amount)` (lines 221-224). `hp` comes from `detail`, which combat-console.tsx:85-93 derives from the `useOptimistic` session (`combatantDetail(session, …)`), and the `adjustEnemyVitals` event "sets one field … to an absolute value" (engine reduce/enemy-vitals.ts). So the absolute wire value is baked from an unconfirmed optimistic frame at click time. Two concrete corruption paths: (1) write A (damage) is in flight when the DM submits write B (heal) — B's absolute embeds A's optimistic effect; if A then fails (network drop, or A loses the commit race and is rejected `stale`), B persists damage the server never accepted, and the optimistic UI reconciles to it, so the failure toast lies. (2) The documented two-DM-tab scenario (use-combat-console.ts JSDoc: "a second DM tab's event"): the ping handler bumps the guard token *before* the refreshed session lands (use-combat-console.ts:99-101 `versionRef.current = ping.version` then `scheduleRefresh()`), so tab B's heal — computed from tab B's pre-refresh `hp.current` — passes the version guard and silently erases tab A's just-persisted damage. The sibling section in the same drawer states the rule this violates: combatant-counters-section.tsx:33-35 "Stepper buttons send a **delta** (±1), never an absolute, so back-to-back taps merge on the server instead of overwriting (the UNN-226 lesson)", matching CLAUDE.md's "Owner-mode writes" convention. The exposure window is narrower than UNN-226's toggles (the AdjustPoolPopover closes per submit), but the ping-window in path (2) lasts until the router.refresh round-trip completes, and the controls are explicitly never disabled (`disabled={false}` at lines 201 and 220), so overlapping writes are easy to issue.

**Fix:** Make the wire payload a delta, not an absolute: extend the event vocabulary with delta semantics (e.g. `adjustEnemyVitals { combatantId, field, delta }` or `damageEnemy`/`healEnemy`), clamp in the reducer against the *server's* fresh row (it already floors at 0; add the max-clamp there too), and keep the same event for the optimistic mirror. Alternatively keep absolutes but compute them server-side from `(field, delta)` inputs. Until then, at minimum wire `isPending` into these popovers instead of `disabled={false}`.

### `apps/web/components/combat/use-combat-console.ts:121-136`

**Back-to-back dispatches within one round-trip both carry the same version token — the second is always rejected "stale", reverts, and shows a misleading toast**

`function dispatch(event) { startTransition(async () => { applyOptimistic(event); const result = await applyCombatEvent({ encounterId: encounter.id, expectedVersion: versionRef.current, event }) … versionRef.current = result.value.version` — `versionRef.current` is captured synchronously at click time and only bumped after the previous response returns. The server guard is a strict compare-and-bump (lib/db/writes/encounter.ts `bumpEncounterVersionGuarded`: `WHERE version = expectedVersion`, else `err("stale")`). So any second event dispatched before the first response (one network round-trip, ~150-500ms) is guaranteed rejected: its optimistic edit appears, then reverts, with the toast "This encounter changed elsewhere. Reload and try again." — wrong on both counts (it was the DM's own tap, and no reload is needed). The hook's own JSDoc claims this is solved ("A rapid follow-up tap … reads the freshly-bumped token synchronously from `versionRef.current` … so the second event isn't spuriously rejected as `stale`") but that only covers taps issued *after* the first response. Natural repros, because none of the drawer controls receive `isPending` (CombatantDrawer passes only `detail`/`onCombatEvent`/`pcVitalsVersions`): ticking two ailments in the conditions popover (it stays open between toggles), tapping a counter's +1 twice (combatant-counters-section.tsx even promises back-to-back taps "merge on the server"), toggling two action-economy chips. The identical pattern is in use-encounter-setup.ts:46-61, whose JSDoc likewise touts "add a combatant, then immediately place or engage it" — yet ImportPcsPanel buttons, ZonesPanel, and CombatantSetupRow are rendered without their `disabled` prop (encounter-setup.tsx:180-247), so two quick setup edits hit the same guaranteed rejection.

**Fix:** Serialize dispatches against the token: keep a promise chain (or queue) in the hook so each `applyCombatEvent` call reads `versionRef.current` after the prior write settles (events are already order-independent intents reduced server-side), or have the server action retry the guarded save once with the row version it just read. Alternatively disable the drawer/setup controls on `isPending` — coarser, but at least honest.

### `apps/web/hooks/use-encounter-snapshot.ts:86-99, 114-137`

**Snapshot fetch responses applied without a version guard — out-of-order responses regress the watch view (indefinitely in realtime mode)**

The ping side compares versions before fetching (line 107: `if (version === undefined || version <= versionRef.current) return`), but both apply sides are unconditional. `refetch()` (lines 89-94): `.then((next) => { if (unmountedRef.current) return; versionRef.current = next.version; setSnapshot(next); setStale(false) })` — and the poll tick (lines 121-126) is identical. There is no `next.version <= versionRef.current` check and no AbortController, so two concurrent fetches whose responses land out of order regress both the rendered snapshot and `versionRef` to the older state. Concretely: two DM writes in quick succession publish pings v5 then v6; ping v6 arrives while the v5-triggered refetch is still in flight (versionRef is still v4, so 6 > 4 triggers a second refetch); if the v6 response lands first, the late v5 response then overwrites it — snapshot and versionRef both regress to v5. The same race exists between `onReconnect: refetch` and a ping refetch, and between overlapping poll ticks when latency exceeds the 1.5s interval (setInterval does not wait for the previous fetch). In polling mode the next tick self-heals in ~1.5s; in realtime mode no poll is running and the next ping only arrives on the next DM write — players watch stale HP/conditions until the DM does something else. Note `versionRef` also regressed, so even a redelivered v6 ping would pass the line-107 guard only because 6 > 5 — but Ably delivers each ping once, so in practice nothing re-triggers.

**Fix:** Add the apply-side mirror of the line-107 guard in both `.then` branches: `if (next.version < versionRef.current) return` (or `<=` plus still clearing `stale`) before `versionRef.current = next.version; setSnapshot(next)`. Versions are monotonic server-side, so dropping older-or-equal responses is always safe. An AbortController keyed to the effect would also close it but the one-line compare is sufficient and matches the existing ping-side idiom.

### `apps/web/hooks/use-character.tsx:255-270`

**Click-write transitions don't catch thrown action errors — a network failure bypasses the toast path and propagates to the error boundary**

`useCharacterWrite`'s `write` runs `startTransition(async () => { if (edit) applyEdit(edit); const result = await dispatchCharacterWriteWithRetry({...}); ... toast.error(...) })` with no try/catch. `dispatchCharacterWriteWithRetry` (hooks/dispatch-character-write.ts:59, 69, 73) awaits the Server Action and `getCharacterVersionsAction` bare — a Server Action invocation *rejects* (rather than returning `Result.err`) on network drop, server crash, deploy-version skew ("Failed to find Server Action"), or an auth interrupt. In React 19, an error thrown in an async transition propagates to the nearest error boundary — so clicking "Take damage" while briefly offline replaces the sheet with the error fallback (or surfaces an unhandled rejection) instead of the designed "Couldn't save. Try again." toast. The sibling debounced path explicitly guards exactly this: use-debounced-auto-save.ts:193-202 catches with the comment "`save` threw (network drop, server crash, auth interrupt)... Roll back, surface a generic toast" — proving the throw case is a known, expected failure mode that the click path simply doesn't handle. The same gap exists verbatim in `useBuilderWrite` (hooks/use-builder-draft.tsx:201-219) and `useOwnCombatEvent.dispatch` (hooks/use-own-combat-event.ts:39-53).

**Fix:** Wrap the awaited dispatch in try/catch inside each transition callback (use-character.tsx, use-builder-draft.tsx, use-own-combat-event.ts), routing a throw to the same generic-error toast the `Result.err` branch uses — mirroring useDebouncedAutoSave's catch. Alternatively centralize it: have `dispatchCharacterWriteWithRetry` catch throws and return a synthetic `Result.err`, so all three callers inherit the fix (useOwnCombatEvent doesn't go through that pipeline and still needs its own catch).

## Themes

Every finding belongs to exactly one theme. Proposed tickets are numbered **T1–T54** for reference.

### [P0] Client-composed post-state writes (UNN-226 class) still shipping

Despite CLAUDE.md documenting the UNN-226 lesson and pointing at combat-state.ts as the worked example, four surfaces still compose the full post-state from optimistic or render-captured values: the ailment editor, the DM drawer's enemy vitals (absolute HP/Max payloads plus client-side clamping), the builder's virtues control (render-time prop adoption clobbers the local draft mid-burst), and the writer sidebar's stale wasActive removal decision. In every case a second interaction within one round-trip silently destroys the first. This is the exact bug class the repo already paid to learn.

- `apps/web/components/character-sheet/combat-state/ailment-editor.tsx:66-155` — Ailment editor composes the full post-state array from optimistic state (UNN-226 class) — concurrent same-field writes are silently clobbered (P0, ✓ verified)
- `apps/web/components/combat/combatant-vitals-section.tsx:196-226` — Enemy vitals controls compose absolute post-state and game-rule clamping client-side instead of delta events clamped in the engine (P2, ⚠ unverified)

## Architecture & DX

### Optimistic write machinery

This subsystem is in unusually good architectural health for its size: one client write primitive (useCharacterWrite) and one autosave primitive (useCharacterAutoSave) cover essentially every owner control on the sheet, both funnel through a single retry-and-broadcast dispatcher, every server write composes one version-guard primitive that also owns the realtime ping, and the optimistic frame is produced by the same pure engine reducer the server uses — so the classic optimistic-UI failure mode (frame disagrees with persisted truth) is structurally prevented rather than tested away. The surface→class concurrency mapping is a compile-checked single source of truth shared by both layers. Bespoke writes are few, enumerated, and (with one exception, RestDialog) individually justified: level-up is genuinely cross-class, the exhaustion clamp is genuinely SQL-shaped, and the watch view's combat-event path writes a different aggregate entirely. The real tax is not design but volume: the per-field action layer is ~65 near-identical 4-step functions and the marginal writable field costs ~6 files, of which roughly three are pure template — the strongest candidate for a small in-house collapse, which the codebase's own 'earn the layer' rule for mechanics already prescribes. The README that anchors the pattern has drifted from the post-engine-extraction reality in several load-bearing examples. The design will carry the next 25 tickets; the findings below are about making each of those tickets cheaper and keeping the two acknowledged forks (builder provider, click-vs-autosave queueing) from drifting silently.

**Strengths:**

- The version-guard choke point genuinely chokes: bumpCharacterVersionGuarded (lib/db/writes/version-guard.ts) fuses the conditional UPDATE, the atomic class increment, the stale/not-found disambiguation, AND the realtime ping into one primitive, and the only two writes that step outside it (the SQL-clamped exhaustion adjust, the dual-class level-up) still compose its exported fragments (characterVersionIncrement, staleOrMissing) and are individually documented — the bespoke set is enumerable and closed.
- EDIT_SURFACE_CLASS (lib/db/version-classes.ts) is a model single-source-of-truth: the grouped-intersection `satisfies` makes the surface→class map both exhaustive and class-correct at compile time, and because client hooks and server wrappers read the same constant, the two layers provably cannot disagree — the failure mode the module exists to remove is actually removed.
- The optimistic frame is structurally honest: useOptimistic applies edits through reduceCharacter — the same engine derivation the server loader runs — so a stat-affecting edit re-derives attributes/affinities/affordability in the same frame, and the exhaustive switch in reduce-character.ts means a new CharacterEdit kind is a compile error until it is both sliced and routed. No game logic leaks into the UI; controls dispatch declarative edits.
- Failure handling is layered and consistent: one silent stale-retry with an explicit budget of one (dispatch-character-write.ts), default toasts with per-call overrides, automatic optimistic rollback via the transition, and a README failure-mode table that matches the code. The dual remote-change transports (BroadcastChannel + Ably) funnel through a single version-compare so the writer's own tab never double-refreshes.
- The UNN-226 lesson is institutionalized, not just remembered: per-field server actions (setBattleConditionAxisAction/setBattleConditionFlagAction) with server-side read-merge-write under the version condition make the back-to-back-clicks lost-update structurally impossible, and CLAUDE.md cites the worked example.
- Uniformity in practice is high: of every owner control audited (pools, victories, combat-state x6, prisma, all five mechanics, inventory + currency, talents, virtues/spark, archetype switcher, inheritance slots, atlas unlock/rank-up, portrait, cast, plus the name/pronouns/narrative autosave fields), all but RestDialog and the justified LevelUpDialog ride the same two provider-bound primitives — and the command palette reuses the identical write path rather than opening a new one.

Parallel forks: builder writes ride hooks/use-builder-draft.tsx (useBuilderWrite/useBuilderAutoSave) over the same dispatch/token/debounce primitives, with leaf-local optimism instead of the central reducer; the watch view's condition edits ride hooks/use-own-combat-event.ts into the encounter aggregate (lib/actions/encounter/own-events.ts) — a different entity, different version token, deliberately outside this pattern.

**Findings:**

- **(P2)** Every per-field Server Action restates the identical 4-step body — Zod safeParse → requireOwner → call the db wrapper → revalidateCharacter on ok — with only the schema, wrapper, and arg-threading varying. Over the next 25 tickets this is the single largest recurring cost in the subsystem.

(2) Apply the codebase's existing mechanics-collapse rule to single-column patches: when a wrapper is just bumpCharacterVersionGuarded + patch (setAilments, setBattleConditions, awardVictories' core), let the action compose the guard inline as the mechanics actions already do (lib/actions/mechanics/knight/valor.ts). The flat-vs-folder contradiction means a new character action has no unambiguous home.
  - *Recommendation:* One doc-sweep pass: fix the four stale references, restate the client pattern §2 against use-inventory-editor.ts/useCharacterWrite, and either soften the aggregate-folder rule to 'character actions stay flat until the migration ticket lands' or actually open that ticket. Cheap, high leverage for an AI-paired workflow where the README is re-read every ticket.

Two same-class writes fired within one round-trip window (e.g. a header HP adjust then a battle-condition toggle, both vitals-class) both read the same versionRef value; the second is rejected stale, then burns its single silent retry (dispatch-character-write.ts:67-80: refetch version, retry once) — three extra round trips. A third overlapping same-class write can stale twice and surface a spurious 'Couldn't sync' toast to a user who conflicted only with themselves. The autosave path solved exactly this with the UNN-274 promise chain; the click path relies on the retry instead.
  - *DX impact:* Self-healing in the common case, so the cost is latency plus a rare false-conflict toast during combat-tempo clicking — and a debugging head-scratcher when someone eventually reproduces the toast and finds the retry budget, the queue that only autosave uses, and the comment 'two writes in the same class still race (correctly)' which was written about third-party conflicts, not same-tab self-races.
  - *Recommendation:* Low-cost hardening when it first bites: chain useCharacterWrite's dispatch through the same `editor.saveQueues[class]` the autosave path uses (the ref already lives on the shared editor context), so same-class clicks serialize and each reads the freshly-bumped token. Keep the retry as the cross-tab/third-party net it was designed to be. Until then, document the asymmetry where the queues are declared.
  - *Cost-benefit skeptic:* The observation is accurate (use-character.tsx:255-262 dispatches directly; saveQueues thread only into autosave at 194-199), but the recommended migration fails cost-benefit. useCharacterWrite is the single highest-traffic owner-write primitive (20+ consumers: header HP, every combat-state toggle, mechanics steppers, skills) and uses useTransition + React auto-revert, whereas the autosave queue is plain promise-chaining with manual rollback (use-debounced-auto-save.ts:167-210) — so chaining the click dispatch through saveQueues is not a copy-paste of the autosave fix but subtle concurrency rework (gating queued work inside a transition while preserving per-control pending and not deadlocking on mid-chain failure), landed on the app's busiest write seam where a regression like a never-draining queue would block all clicks. The disease is rare and self-healing: it needs two same-class writes inside one ~few-hundred-ms round-trip (deliberate popover/Select interactions), the single silent retry resolves the common case, and the spurious toast needs a third overlapping same-class write — no ongoing tax on new features. For a single-dev, one-ticket project the recommendation's own cheaper fallback (document the asymmetry where the queues are declared) is the proportionate fix; the full migration is cure-worse-than-disease.
- **(P3)** The sheet provider and the builder provider are deliberate near-clones: useBuilderWrite (hooks/use-builder-draft.tsx:183-223) duplicates useCharacterWrite's transition + dispatch + onError/toast body (hooks/use-character.tsx:239-274) line-for-line modulo two divergences (callback-style `optimistic` instead of a reducer `edit`; an extra `onSuccess`), and useBuilderAutoSave/useCharacterAutoSave are the same wrapper twice. The fork is documented and reasoned (use-builder-draft.tsx:32-41: the builder holds a raw incomplete row, not a HydratedCharacter, so coupling to reduceCharacter would be wrong), and both forks share the real primitives (dispatchCharacterWriteWithRetry, useCharacterTokenRef, useDebouncedAutoSave). But pipeline changes now have two application sites: the UNN-274 queue threading already had to be mirrored into both providers, and the builder write lacks the sheet's queue-less click path improvements and vice versa (sheet write has no onSuccess; the portrait upload JSDoc in use-character.tsx:207-212 explicitly borrows the builder's shape).
  - *DX impact:* Every enhancement to the write pipeline (queueing clicks, richer error routing, instrumentation) costs a second mirrored edit, and the two hooks' option bags are already drifting apart (onSuccess exists on one side only), which makes 'which hook am I extending?' a real question for new surfaces that straddle builder and sheet (the portrait control exists in both trees).
  - *Recommendation:* Don't merge the providers — the HydratedCharacter/raw-row split is real. If a third divergence-fix lands, extract just the shared transition body (dispatch → onError-or-toast, ~25 lines, parameterized by an optional pre-dispatch callback and optional onSuccess) into dispatch-character-write.ts next to the retry helper, leaving each provider a thin binding. Until then this is an accepted, documented duplication — flagging so the mirroring stays deliberate.

### Realtime & invalidation stack

This is one of the cleanest subsystems in the app: the ADR's invalidation-ping design is implemented almost exactly as written, and the three load-bearing claims all verify — channels.ts is provably the only place channel names are assembled (publish helper + token route are the only consumers of realtimeChannelName), degradation without ABLY_API_KEY is a genuine whole-layer no-op (null REST client → silent publish, 503 token route → inert hooks → poll/BroadcastChannel/navigation fallbacks), and the wire never carries domain data, so the server-side enemy redaction cannot be bypassed by transport. There is a real reusable core, not two bespoke stacks: useRealtimeChannel (+ its RealtimeChannelListener form) serves all four surfaces, and a third invalidation-shaped surface (e.g. campaign-roster invalidation) would cost roughly a domain literal in channels.ts, an enum entry in the token route, a ~10-line publish helper, and a parse function. What is bespoke per surface — deliberately, but increasingly expensively — is the receive-side version-token bookkeeping: five hand-rolled variants of "ref synced from server prop + compare + refresh" with inconsistent forward-only semantics, which is exactly what the already-filed UNN-374 should absorb when the next surface lands. The two genuine asymmetries worth fixing are publish-side: character pings are structurally unforgettable (inside the version-guard) while encounter pings are per-action discipline, and addSetupCombatantsAction already proves the discipline fails — it bumps the encounter version without pinging. Note that true presence (the hypothetical "roster presence") would not ride this core cheaply: the primitive deliberately exposes only onPing over per-channel connections, so presence requires UNN-376's shared-connection work plus extending the hook's API — a known, ticketed cliff rather than a surprise. Remaining findings are narrow, self-healing failure modes (one-shot token fetch per mount, out-of-order snapshot fetches, phantom ping on transaction rollback) appropriate to flag but not to pre-build around.

**Strengths:**

- channels.ts genuinely is the single naming authority — verified by grep: realtimeChannelName is referenced only by publish.ts, the token route, and its own test. Clients receive the server-resolved name from the token response (use-realtime-channel.ts:117, route.ts:38-43) and never see the namespace; the preview-isolation property is enforced by construction, not convention, and channels.test.ts pins the whole env matrix.
- Degradation is zero-config and total: getAblyRest() returns null without ABLY_API_KEY (lib/realtime/client.ts), which makes publish a silent no-op, the token route a 503, and every subscriber inert — the watch view's poll, the sheet's BroadcastChannel, and navigation freshness remain as documented fallbacks. Local dev and E2E require nothing, exactly as ADR Decision 3/7 promised.
- The advisory-ping discipline holds at every publish site: payloads are only {versions} or {version, status}, never domain data, so the UNN-324 server-side enemy redaction is structurally unbypassable over the realtime channel, and echo suppression falls out of the version compare with zero sender-id plumbing.
- Character-side publish lives at the true choke point (inside bumpCharacterVersionGuarded), so ~57 guarded write call sites get the ping for free and the two bespoke UPDATEs compose the same characterVersionIncrement — a new character write cannot forget to ping.
- One subscribe primitive (useRealtimeChannel) serves all four surfaces, with the RealtimeChannelListener mountable form turning dynamic channel lists (per-PC, per-encounter) into plain React mount/unmount — no imperative attach/detach bookkeeping anywhere. The lazy `await import("ably/modular")` inside the effect keeps signed-out bundles clean.
- The two character transports (Ably ping, UNN-203 BroadcastChannel) funnel through one shared compare (mergePingedVersions in hooks/character-version-sync.ts), so dual delivery never double-refreshes a tab — the exact failure mode dual transports usually breed.
- The ADR is a model document: every implementation deviation is amended inline with its ticket (one app not three, no ably/react, the banner's all-non-ended subscription), and deferred work (UNN-374/375/376) has explicit triggers rather than schedules. The code matches the doc to an unusual degree.

**Paved-path trace ("battle cry" field):**

SHEET PATH (write→ping→refresh): 1) Owner edit dispatches via hooks/use-character.tsx (useCharacterWrite → dispatchCharacterWriteWithRetry in hooks/dispatch-character-write.ts, which also fires the cross-tab BroadcastChannel on success via hooks/use-character-versions-broadcast.ts). 2) The Server Action's write composes lib/db/writes/version-guard.ts:bumpCharacterVersionGuarded, which on success calls publishCharacterPing (lib/realtime/publish.ts) — two bespoke writes (lib/db/writes/combat-state.ts:173 exhaustion, lib/db/writes/leveling.ts:169 level-up) publish directly. 3) publish.ts schedules the POST via next/server after(), through the lazy Ably.Rest singleton (lib/realtime/client.ts), onto the name from lib/realtime/channels.ts ({ns}:{domain}:{shortId}; ns from VERCEL_ENV). 4) Subscribers: CharacterProvider (hooks/use-character.tsx:142-150) mounts hooks/use-realtime-channel.ts, which POSTs {domain, shortId} to app/api/realtime/token/route.ts (503 when no key; otherwise server-resolved channel + subscribe-only TokenRequest), lazily imports ably/modular, and subscribes. 5) Both the Ably ping (parseCharacterPing) and the BroadcastChannel message funnel through mergePingedVersions (hooks/character-version-sync.ts) against the per-class refs (hooks/use-character-token-ref.ts); anything fresher → router.refresh(). DM-console branch: components/combat/combat-console.tsx mounts RealtimeChannelListener per PC + use-combat-console.ts subscribes the encounter channel; pings route through pc-ping.ts:decidePcPing / an inline version compare into a microtask-coalesced refresh. Campaign banner: components/campaign/encounter-status-listener.tsx, one listener per non-ended encounter, refresh on status change only. **Findings:**

- **(P2)** The two domains have different structural guarantees that a write actually publishes its ping. Character pings fire inside the shared guard primitive (`bumpCharacterVersionGuarded` in /Users/jackson/Developer/unnamed-system/apps/web/lib/db/writes/version-guard.ts:102 — its JSDoc: "the single choke point... so the ping can't be forgotten per write"), covering ~57 call sites structurally, with the two bespoke UPDATEs (writes/combat-state.ts:173, writes/leveling.ts:169) re-spelling it. Encounter pings, by contrast, are per-action discipline in the action shells (lib/actions/encounter/events.ts:98, end.ts:53, own-events.ts:70) — the underlying guard `bumpEncounterVersionGuarded` (lib/db/writes/encounter.ts:94) does not ping. And one encounter write already missed it: `addSetupCombatantsAction` (lib/actions/encounter/setup.ts) saves the session and bumps `version` with no `publishEncounterPing`. A catalog enemy-add advances the version silently; any subscribed peer (a second DM tab, the draft watch view) holds a now-stale token until the next pinging write heals it, and a write issued from that peer takes a spurious stale rejection.
  - *DX impact:* Every new encounter write path (and there will be more — the combat console is the active feature area) must remember to add the ping by hand; the existing codebase already contains one counterexample to copy from. Forgetting produces the worst kind of bug: no error, just a surface that stops being live for one class of edit, discovered during a play session.
  - *Recommendation:* Move the encounter ping to the same depth as the character one: have `bumpEncounterVersionGuarded` `.returning` the `shortId` and `status` alongside `version` and publish there (mirroring version-guard.ts). The shells that need a different `status` in the payload (startCombat's flip) can keep their explicit publish; the guard's becomes the safety net. At minimum, add the missing `publishEncounterPing` to `addSetupCombatantsAction`.
- **(P2)** Version-token bookkeeping — the receive half of the protocol — is hand-rolled five times with inconsistent semantics: `useCharacterTokenRef` (unconditional assignment gated by `[token]` deps, hooks/use-character-token-ref.ts), `useCombatConsole`'s encounter `versionRef` (unconditional effect sync, components/combat/use-combat-console.ts:63-66), the same hook's `pcVitalsVersions` (forward-only max, lines 74-82, with a comment explaining why regression must be prevented), `useOwnCombatEvent`'s ref (forward-only, hooks/use-own-combat-event.ts:31-36), and `useEncounterSnapshot`'s `versionRef` (unconditional set from fetch results, hooks/use-encounter-snapshot.ts:69, 91, 122). Each variant's monotonicity reasoning lives in its own comment, and whether forward-only protection is needed is re-derived per site. The compare/refresh policies are similarly bespoke: `mergePingedVersions`, `decidePcPing` (components/combat/pc-ping.ts), the inline `<=` compares in use-encounter-snapshot.ts:107 and use-combat-console.ts:99-101, and the status-only compare in encounter-status-listener.tsx:38-44. The ADR already files this as UNN-374 ("consolidate version-token handling into a first-class client primitive", trigger: the next surface that needs tokens).
  - *DX impact:* Building the next realtime surface means hand-rolling a sixth variant of "ref synced forward from a server prop + compare + dedup'd refresh" and re-litigating the forward-only question in review. The subtle inconsistencies are also where the next race bug lives — the unconditional-sync variants rely on dep-gating arguments that the forward-only variants explicitly reject.
  - *Recommendation:* Honor UNN-374's trigger exactly as written: when the next token-consuming surface appears, extract one ~30-line primitive (forward-only version ref + ping compare + microtask-coalesced refresh, generalizing `scheduleRefresh` from use-combat-console.ts:84-92) and migrate the five sites opportunistically. Don't build it speculatively — the ticket's trigger discipline is correct for a solo project.
- **(P2)** The realtime path's availability is decided once per mount: in `useRealtimeChannel` (hooks/use-realtime-channel.ts:110-114), if the initial token POST fails — a transient network blip at page load, a cold-start 503, a flaky hotel wifi at the game table — the hook reports unavailable and stays inert for the component's lifetime. The false→true availability transition only exists via the Ably SDK's own reconnection, which requires a constructed client, which requires that first fetch to have succeeded. The watch view then polls at 1.5s forever (functional but exactly the churn the ADR built realtime to avoid); the character sheet and DM console silently lose cross-user sync until a remount.
  - *DX impact:* A single failed request at mount permanently downgrades a multi-hour game session. Hard to diagnose because everything still works — just at navigation-freshness instead of live — so it presents as "realtime is flaky" with nothing in the console. Every surface inherits this cliff from the one hook.
  - *Recommendation:* Add one bounded re-attempt path inside `useRealtimeChannel` — e.g. retry the initial token fetch on `online`/`visibilitychange` or after a single delayed timer. One change in one file upgrades all four surfaces; no per-surface code.
- **(P2)** `useEncounterSnapshot` applies fetch responses unconditionally: both the ping-triggered `refetch()` (hooks/use-encounter-snapshot.ts:86-99) and the poll tick (lines 118-131) do `versionRef.current = next.version; setSnapshot(next)` with no version compare on the response. Two in-flight fetches (a ping-triggered one racing a poll tick during fallback flapping, or two pings arriving in separate tasks) can resolve out of order, displaying an older snapshot and regressing `versionRef` below the version already shown. The next ping/poll heals it, but for up to one cycle the watch view can visibly step backward (HP undoing, a turn un-advancing).
  - *DX impact:* A confusing, unreproducible flicker during live play — the kind of report that burns an evening of debugging because the code reads correct and the race needs two overlapping fetches to manifest.
  - *Recommendation:* One guard in the shared apply path: drop a response whose `version <= versionRef.current` (mirroring the ping compare at line 107). Extract the duplicated then/catch body of refetch and the poll tick into one function while there.
- **(P3)** Character pings can fire for writes that never committed. `bumpCharacterVersionGuarded` publishes from inside the caller's transaction executor — e.g. `replaceInventory` (lib/db/writes/inventory.ts:72-109) runs the guard first inside `db.transaction`, then per-row deletes/inserts that can throw and roll back the bump. The publish is scheduled via `after()` (lib/realtime/publish.ts:58-64), which runs when the response finishes regardless of commit vs rollback; publish.ts's JSDoc reasons about commit ordering ("after any wrapping db.transaction has committed, so a ping can't race its own refetch") but not the rollback case. A phantom ping forwards every subscriber's refs past the committed version (mergePingedVersions) and triggers a refresh; those tabs' next write of that class is then guaranteed a stale rejection and silent-retry round-trip.
  - *DX impact:* Self-healing (the retry pipeline in hooks/dispatch-character-write.ts recovers) and rare, but it quietly breaks the documented invariant that pings are post-commit, which a future maintainer will rely on when reasoning about the next surface.
  - *Recommendation:* Document the rollback caveat in publish.ts/version-guard.ts now; only restructure (return the ping payload from the guard, publish after the transaction resolves at the shell) if a real symptom appears — restructuring trades away the can't-forget-the-ping property, which is worth more than this edge case.
- **(P3)** The wire contract has no neutral home and no schema. `CharacterPing`/`EncounterPing` are defined in lib/realtime/publish.ts, which is `import "server-only"`; the client-side parse helpers import them type-only (hooks/character-version-sync.ts:6, hooks/encounter-ping.ts:3 — safe because type imports are erased, but one accidental value import breaks the client build). The receive side then re-derives the shape by hand-rolled narrowing (`parseCharacterPing`, `parseEncounterPing`) rather than the Zod schemas the rest of the stack uses for every other wire boundary, so publisher and parser can drift with no compile-time or test-time tripwire connecting them.
  - *DX impact:* Adding a field to a ping (e.g. the next surface wants `{domain, id, version, actor}`) means editing the interface in a server-only file and remembering to mirror it in a hand-written narrowing function elsewhere; nothing fails if you forget — the field just silently parses out.
  - *Recommendation:* When the third domain lands, move the ping payload types + a Zod schema per domain into a small shared module (lib/realtime/pings.ts, no server-only), with publish.ts and the parse helpers both importing it; the parse helpers collapse to `schema.safeParse`. Not worth a standalone ticket before then.

### Data loading & hydration architecture

This subsystem is in good architectural shape and will carry the next 25 tickets. The spine — impure fetch (load-character.ts) feeding a pure, deps-curried deriveHydratedCharacter, with the identical function powering the client's optimistic reduceCharacter — is the strongest pattern in the app: server and optimistic states cannot structurally diverge, and the engine seam (RawCharacterInputs in, HydratedCharacter out, gameData bound once in lib/game-engine.ts) is clean and well-documented. The provider granularity question resolves in the design's favor: yes, all ~32 useCharacter() call sites re-render on any optimistic edit (one context object, new identity per reduce pass), but that is the point of UNN-237 — stat edits cascade into attributes/affinities/skill costs, so slice-level subscription would re-derive the same values anyway — and React Compiler bounds the cost at this sheet's scale; no action warranted. Loading patterns are mostly consistent (server page → single serialization → client provider; lean summary projections for lists), with two genuine drifts: encounter surfaces assemble PC detail maps inline in route bodies (duplicated between the console page and the snapshot loader), and per-request memoization is ad-hoc per route rather than query-layer cache(), which already produces measurable duplicate loads on the watch page plus a comment asserting a cache that does not exist. Redaction is structurally guaranteed exactly where it must be — the enemy arm of the snapshot type has no attribute/affinity keys and both the page and poll API flow through the one projector — but the parallel "lean slice" claim on the DM console's PcCombatantDetail is type-level fiction: the runtime spread ships each PC's full sheet (skills, inventory, notes) to the console client, a payload-bloat issue today and a latent privacy trap the moment any owner-only field lands on CharacterRow.

**Strengths:**

- The fetch/derive split is genuinely load-bearing: lib/db/queries/load-character.ts owns I/O (fetchRawInputs, four concurrent child queries), and the pure deriveHydratedCharacter (packages/game/src/engine/character/derive-hydrated-character.ts) is the same function the client's optimistic reducer re-derives through (reduceCharacter in CharacterProvider). Optimistic frames cannot structurally drift from the server's, and "no game logic in the UI" is enforced by construction, not review.
- HydratedCharacter as the single lingua franca works: foundation owns the row contracts (records.ts) with a conformance test pinning the Drizzle tables, the loader is the only constructor, every surface only consumes. PcCombatantDetail and the watch column reuse it (the same SheetHeader/Affinities/MechanicWidget/Skills render on /c/{shortId} and inside the encounter watch left column via the same CharacterProvider) — zero forked sheet rendering across surfaces.
- Enemy redaction is structurally guaranteed, not convention-guaranteed: PlayerVisibleCombatant's enemy arm (packages/game/src/engine/encounter/player-snapshot.ts) simply has no attributes/affinities keys, the projector never writes them, and the public page and the poll API share the one getEncounterSnapshot seam — deleting the watch view cannot re-expose anything. This is the right way to do it.
- loadOwnedEncounterSheets decides ownership on the cheap character row before hydrating, so another player's full sheet is never even loaded server-side, let alone serialized — privacy enforced at the loader, with the reasoning documented inline.
- RSC/client boundaries are in the right place for the product: pages do auth + load + status forks server-side and serialize exactly once into a client provider; list surfaces (CharacterSummary, EncounterSummary in character-list.ts/load-encounter.ts) are deliberate lean single-round-trip projections that never pull the session blob or child rows.
- The two-context split (CharacterContext for reads, CharacterEditorContext for writes) with reactCompiler: true keeps write-only consumers referentially stable, and the builder mirrors the same provider pattern with its divergence from CharacterProvider explicitly argued in use-builder-draft.tsx — a paved path with documented exits.
- The realtime/freshness loop composes with loading instead of duplicating it: pings are advisory version numbers, and both the sheet (router.refresh through the shared version-compare) and the watch view (refetch through the same snapshot API) re-enter the existing server read path — one read path per surface, no second wire format.

**Findings:**

- **(P2)** The DM combat console ships each PC's entire hydrated sheet to the client, while the type and its documentation claim a lean slice. In apps/web/app/combat/[shortId]/page.tsx (live branch, lines ~94-110) pcDetailById is built as `{...c, className}` over each full HydratedCharacter and passed as a prop to CombatConsole, which is a "use client" component (apps/web/components/combat/combat-console.tsx line 1) — so the runtime object serialized into the RSC payload contains every field: all hydrated skills (each with resolvedCost/resolvedAttackRoll), full inventory, knives, chains, ancestryText, backgroundText, and notes markdown (fields confirmed on CharacterRow in packages/game/src/foundation/character/records.ts lines 68-77). PcCombatantDetail is a `Pick<HydratedCharacter, ...>` (packages/game/src/engine/encounter/roster-view.ts line 54), and its JSDoc states "The client payload still skips the skills/inventory/child rows the console never renders" — but Pick narrows the type only; the spread copies everything and TypeScript's excess-property check does not apply to spread properties. The same full-spread pattern exists in lib/db/queries/load-encounter-snapshot.ts lines 47-59, where it is harmless today only because projectPlayerSnapshot explicitly picks fields server-side.
  - *DX impact:* Console payload grows with party size and sheet content for data the rail/drawer never renders. More importantly, the contract lies: a developer adding any owner-private or sensitive field to CharacterRow (the moment one ever exists) will read the PcCombatantDetail doc, believe the boundary trims, and ship it to the DM client — and potentially copy the same spread into a player-facing surface. The redaction story is structurally airtight at the snapshot seam and silently false at this one.
  - *Recommendation:* Add a small explicit mapper (e.g. toPcCombatantDetail(character): PcCombatantDetail that lists the ~13 fields) and use it at both assembly sites (the console page's live branch and load-encounter-snapshot.ts). Fix the roster-view.ts JSDoc to match whichever behavior is chosen. This is a one-function change that makes the documented guarantee true.
- **(P2)** The PC-detail assembly block (flatMap PC characterIds → hydrate each via loadHydratedCharacterById → build a characterId-keyed map with className resolved via getArchetype) is duplicated nearly verbatim between the DM console route body (apps/web/app/combat/[shortId]/page.tsx lines 87-110) and the snapshot loader (apps/web/lib/db/queries/load-encounter-snapshot.ts lines 38-59), with a third hand-rolled variant (pcStatsById, Agility/Luck only) in the same page's draft branch (lines 56-77). The console page also imports getArchetype directly from @workspace/game/data, bypassing the lib/game-engine.ts composition root the rest of the app routes catalog lookups through. CLAUDE.md's own convention says per-surface data shaping should live next to the data, not in the component/route body.
  - *DX impact:* Every new encounter surface (and the UNN-3xx backlog keeps producing them) re-copies this block; a change to PcCombatantDetail or to how className resolves must be hand-synced across two-plus sites, and the console page silently drifting from the snapshot loader is exactly how the watch view and DM view would come to disagree about a PC. ~50 lines of data assembly inside a route component also makes the status fork harder to scan.
  - *Recommendation:* Extract one loadPcDetailById(characterIds: string[]) helper into lib/db/queries (beside load-encounter-snapshot.ts), used by the console live branch and getEncounterSnapshot; have the draft branch's pcStatsById derive from its result or stay as the one bespoke case. Route the className lookup through the composition root while there. Pairs naturally with the explicit-picker fix from the payload finding.
- **(P2, recommendation contested)** Per-request memoization is applied route-by-route instead of at the query layer, and it has already drifted: no file in apps/web/lib/db/queries/ uses React cache() (verified by grep), while four separate route-level wrappers exist (getCharacter in app/c/[shortId]/page.tsx, getSnapshot in app/c/encounter/[shortId]/page.tsx, getEncounterForDM in app/combat/[shortId]/encounter-access.ts, getBuilderCharacter in app/builder/[shortId]/_loader.ts) and the atlas page calls loadHydratedCharacterByShortId bare. The watch page's comment (app/c/encounter/[shortId]/page.tsx lines 18-19) claims "its loadEncounterRowByShortId read is itself React-cached" — it is not (lib/db/queries/load-encounter.ts imports no cache). Concretely, one watch-page render loads the encounter row twice (once via getEncounterSnapshot, once via loadOwnedEncounterSheets), reads each owned PC's character row three times (the ownership check in loadOwnedEncounterSheets, plus inside loadHydratedCharacterById, plus inside getEncounterSnapshot's hydration), and runs the full 5-query hydration of each owned PC twice (snapshot pcDetail + ownedSheets).
  - *DX impact:* Every new surface must remember to hand-roll its own cache wrapper or silently pay duplicate loads; composite loaders that reuse the shared primitives (the paved path!) multiply round trips invisibly. The stale comment shows the convention is already believed to be something it isn't — the next composite loader will be written on that false assumption.
  - *Recommendation:* Wrap the shared row/hydration loaders once at the query layer — loadCharacterRowById/ByShortId, loadHydratedCharacterById/ByShortId, loadEncounterRowByShortId — in React cache() (they are id-keyed pure reads, the textbook case), then delete the now-redundant page-level wrappers and the false comment. One small PR; every existing and future composite loader gets request-level dedup for free.
  - *Cost-benefit skeptic:* The observation is accurate (no cache() in queries/, the comment is genuinely false, the watch page does double-load), but the recommendation is the wrong shape and net-negative. The shared loaders (loadHydratedCharacterById/loadCharacterRowById) are dual-use: called from 5 write modules (rest/adjust-pools/cast-skill/combat-state/leveling) as well as RSC pages, all sitting atop the UNN-248 optimistic-concurrency spine that depends on fresh reads — so wrapping them in cache() at the query layer plants a latent staleness footgun in the correctness-critical write path (a future read→mutate→re-read would silently get stale data), strictly worse on the very "convention believed to be something it isn't" axis the finding raises. The actual recurring benefit is tiny: only one page (the watch page) truly double-loads; the combat page calls the loader N times for N distinct PCs (cache does nothing), and the atlas page is a separate route render with nothing to dedup against. The page-level cache() wrappers are the idiomatic Next seam (generateMetadata + body share a request) and would survive a query-layer change anyway; the correct fix is local — repair the stale comment and, if the watch double-load matters, cache() only the read-only loadEncounterRowByShortId primitive or pass the encounter row down within that page.
- **(P2, recommendation contested)** The public, unauthenticated snapshot endpoint performs full character hydration per request for a vitals-slice read. getEncounterSnapshot (lib/db/queries/load-encounter-snapshot.ts) calls loadHydratedCharacterById for every PC combatant — each is 1 row query + 4 child-row queries + the full deriveHydratedCharacter pass — only for projectPlayerSnapshot to keep name, HP/SP, attributes, and portraitUrl. The route (app/api/encounter/[shortId]/snapshot/route.ts) has no auth, no Cache-Control, and no coalescing; in the documented polling fallback (use-encounter-snapshot.ts, POLL_INTERVAL_MS = 1500) each viewer triggers roughly 2 + 5N queries every 1.5 seconds for an N-PC encounter.
  - *DX impact:* Runtime rather than authoring cost: a 4-PC table with a few spectators in degraded mode is ~22 queries per poll per viewer against Neon, and the endpoint is an anonymous request-amplification point (one cheap GET fans out ~22 queries plus engine derivation). Healthy Ably reduces this to ping-triggered refetches, so today it is tolerable for one table — but every future watch feature (and any public sharing of watch links) inherits the per-poll fan-out.
  - *Recommendation:* Proportionate fix, not a redesign: keep the full-hydration reuse (it is the paved path and guarantees consistency with the sheet), but add a ~1s coalescing layer on the route — unstable_cache keyed by shortId or an equivalent micro-cache — so concurrent pollers share one assembly. Build a lean vitals-slice loader only if watch traffic actually grows.
  - *Cost-benefit skeptic:* The observation is accurate (confirmed: getEncounterSnapshot runs full hydration — 1+4 queries + deriveHydratedCharacter per PC — only to fill a lean Pick<> vitals slice; the route has no auth/Cache-Control/coalescing). But the recommendation fails the cost-benefit lens for this single-dev, one-table project. Per docs/realtime/ADR.md the per-poll fan-out only occurs in the DEGRADED (Ably-down) mode, and only when multiple viewers watch the SAME encounter concurrently — and the ADR documents ≤10 concurrent connections total with ~400x Ably headroom, so that intersection is near-zero in practice. Worse, the recommended cheap fix is itself risky: an unstable_cache keyed only by shortId with a ~1s TTL would serve a stale snapshot on ping-triggered refetches, directly undermining the realtime freshness contract that the ping path exists to guarantee; making it correct requires version-keyed tag invalidation wired into the write shells, which is no longer the "thin micro-cache" the recommendation sells (and unstable_cache has zero precedent in this codebase). The cure (latent staleness regression + a new unstable API pattern) is worse than a disease that only manifests under a rare degraded-plus-concurrency intersection.
- **(P3, recommendation contested)** The builder's draft loader reaches into write modules for reads: app/builder/[shortId]/_loader.ts imports loadCharacterKnives from lib/db/writes/knives.ts (line 103) and loadCharacterChains from lib/db/writes/chains.ts (line 100) — read functions living in writes/ files, against the documented lib/db role grouping (queries/ for reads, writes/ for persistence). The BuilderCharacter shape itself (raw row + knives + chains + originArchetypeKey) is a deliberate, well-documented divergence from HydratedCharacter (use-builder-draft.tsx explains why it must not reuse CharacterProvider), so the third hydration shape is fine — it is the file placement that drifts.
  - *DX impact:* Discoverability cost only: a developer hunting for "where do we read knives/chains" checks queries/ first and finds nothing; the next child-row table added to the builder will follow whichever precedent is found first, entrenching the split. Small but it erodes the otherwise-clean queries/writes contract the CLAUDE.md naming rule depends on.
  - *Recommendation:* Move (or re-export) loadCharacterKnives/loadCharacterChains into lib/db/queries — either a small queries/knives.ts + queries/chains.ts pair or folding them into the existing per-concern files — and leave _loader.ts as the route's cache wrapper. Mechanical, low-risk.
  - *Cost-benefit skeptic:* The observation's facts hold (reads exported from writes/knives.ts:103 and writes/chains.ts:100, imported by _loader.ts:12-13), but the recommendation fails cost-benefit. The recommended fix is a one-line re-export pair (queries/knives.ts → loadCharacterKnives), which is exactly the "vestigial indirection / pass-through wrapper that only re-types a delegate" the user's own conventions actively delete; the only alternative — relocating the read away from its four write siblings — splits a cohesive ~30-line table-binding module (knives.ts/chains.ts each bind named-entry-list.ts helpers to one table). Critically, the queries-side precedent the move claims to restore doesn't exist: the central sheet loader queries/load-character.ts inlines its own raw db.select().from(characterChains) rather than calling loadCharacterChains, so the de-facto convention is "child-row reads live next to where they're needed," not "all reads in queries/." With only two callers, a fixed game-data schema (Knives/Chains are the only named-entry tables, new ones are rare), the "recurring tax on every new feature" is overstated — this is a one-time P3 discoverability nit whose cure adds rejected files or worsens cohesion.

### Combat/encounter state containers

The three combat surfaces share one mental model of authoritative encounter state, not three: the `session` jsonb on the encounter row, advanced only by `CombatEvent`s through the single pure `reduceCombatSession`, version-guarded by one `version` token. The setup shell and live console are not two mechanisms — UNN-347 deliberately collapsed setup onto the console's optimistic dispatch path (`useEncounterSetup` and `useCombatConsole` both run `useOptimistic(encounter.session, reduceCombatSession)` and POST the event to `applyCombatEvent`), so the lifecycle fork (draft/live/ended) is a server-side exhaustive switch in `app/combat/[shortId]/page.tsx` over one persistence model, not divergent state containers. The watch view is a second *read projection* of the same state (`projectPlayerSnapshot` → `EncounterSnapshot`, redaction structural in the type), consumed via ping-or-poll refetch in `useEncounterSnapshot`; its player write path (`applyOwnCombatEvent`) reduces the same reducer server-side but cannot mirror optimistically because the client only holds the redacted projection. The localStorage enemy queue is correctly *not* encounter state — a staging cart committed through the same `addCombatant` reducer. The architecture is unusually coherent for its size; the real costs are in the client plumbing, which is copy-pasted rather than factored (three hand-rolled version-ref + dispatch-transition blocks, with diverging sync semantics), and in two freshness seams: the watch page's server-loaded props (`ownedSheets`) never reconcile with the live snapshot, and the snapshot is a cross-aggregate projection (session + character rows) invalidated by a single-aggregate token. Adding a fourth surface means copying ~40 lines of dispatch plumbing and choosing a read model; adding a per-combatant property means a principled but undocumented 6–8 file fan-out.

**Strengths:**

- One write spine, event-as-wire-payload: every roster/zone/turn/overlay edit on every surface is a CombatEvent reduced by the same pure reduceCombatSession on server (lib/actions/encounter/events.ts, own-events.ts, setup.ts) and mirrored optimistically on the DM client (use-encounter-setup.ts, use-combat-console.ts). A client can never persist a computed session, and the optimistic frame is structurally identical to what the server will store. UNN-347 retiring the Save-button/full-roster-POST in favor of this path removed the classic last-write-wins merge bug class before it could recur.
- Status branching is principled, not scattered: each surface forks exactly once at its root — an exhaustive switch on encounter.status in app/combat/[shortId]/page.tsx (setup/console/ended-stub as three components), a redirect guard in the enemies sub-route, and a single snapshot.status conditional at the top of EncounterWatch (which must be client-side because status flips under a parked viewer). No status conditionals leak into leaf components.
- Redaction is structural and lives in one projector: PlayerVisibleCombatant's enemy arm has no attributes/affinities keys at all (packages/game/src/engine/encounter/player-snapshot.ts), so the public snapshot route and any future watch component cannot leak hidden enemy data — the compiler enforces the visibility model.
- Shared presentational kit with per-surface shapers converging on one view shape: resolveZoneLayout (DM) and resolvePlayerZoneLayout (watch) both emit ZoneLayoutView so components/combat/zone-layout.tsx renders both battlefields; ConditionsControls is the single overlay editor behind both the DM drawer (combatant-conditions-section.tsx) and the player's own control (player-combat-state-control.tsx); EnemyStatblock serves browse + drawer. View logic stays out of components per the CLAUDE.md convention.
- The enemy queue is correctly modeled as a cart, not a shadow roster: hooks/use-encounter-enemy-queue.ts is useSyncExternalStore over localStorage (hydration-safe, no stale closures, cross-tab), and commit folds entries through the same addCombatant reducer server-side (lib/actions/encounter/setup.ts), explicitly dodging the merge trap the retired saveEncounterSetupAction had.
- Lifecycle-aware and echo-aware traffic: useEncounterSnapshot suspends subscription and schedules no interval once status is ended; ping version-compares drop echoes of a surface's own writes; the console microtask-dedupes refreshes (use-combat-console.ts scheduleRefresh); polling survives as a silent degraded mode rather than an error state.
- One shared error taxonomy: encounterErrorMessage is the single toast-copy map for every encounter write surface, so phrasing cannot drift between setup, console, browse-commit, and the player control.

**Findings:**

- **(P2)** The version-token-ref + dispatch-transition plumbing is hand-rolled three times in the combat layer with diverging semantics, despite an existing primitive whose JSDoc claims to be "the one place that prop-sync pattern lives" (hooks/use-character-token-ref.ts). use-encounter-setup.ts lines 41–61 and use-combat-console.ts lines 63–66 + 121–136 contain a character-for-character identical 16-line dispatch body (applyOptimistic → applyCombatEvent → toast on error → bump versionRef → router.refresh) plus the same ref-and-effect pair; hooks/use-own-combat-event.ts hand-rolls a third copy whose ref sync is forward-only (`if (snapshotVersion > versionRef.current)`) while the other two assign unconditionally — two subtly different concurrency semantics for the same concept, chosen per-file rather than by a primitive.
  - *DX impact:* Every new combat write surface (player token movement, a co-DM view, a campaign-page quick action) copies the block a fourth time and must re-decide the sync semantics; a fix to one copy (e.g. self-healing on stale, transition batching) won't propagate. The semantic divergence is exactly the kind of drift that produced the UNN-226 cautionary tale on the character side.
  - *Recommendation:* Extract one small hook — e.g. useEncounterDispatch(encounter) returning {session, isPending, dispatch} built on a shared version-token ref (or reuse useCharacterTokenRef for the ref half) — and have setup, console, and own-event compose it; the console keeps its realtime/PC-vitals machinery as additions. Pick one sync semantic (forward-only is the safer of the two) and document why.
- **(P2)** The watch page mixes two data sources with different freshness models and only one updates: the snapshot live-updates via useEncounterSnapshot, but ownedSheets (which decides whether the viewer gets the 3-column layout and their conditions control) is loaded once server-side in app/c/encounter/[shortId]/page.tsx and never reconciled — EncounterWatch contains no router.refresh path, and useEncounterSnapshot only setState()s the snapshot. A player who parks on the "Waiting for the DM to start combat" state (the exact flow the draft branch exists for) before the DM adds their PC to the roster, or whose PC joins mid-fight via addCombatant, watches the whole encounter in spectator layout until they manually reload.
  - *DX impact:* A user-visible wrong-layout state in a mainline table flow, recoverable only by a reload nobody is told to do. It also sets a trap for future tickets: any new server-loaded prop on the watch page (e.g. campaign roster, party info) silently inherits the same never-reconciles behavior while the snapshot beside it stays live.
  - *Recommendation:* In useEncounterSnapshot (or EncounterWatch), call router.refresh() when a refetched snapshot's status differs from the previous one, and optionally when the set of pc combatant ids changes — that re-runs the server loaders at exactly the moments ownedSheets can change. Cheap, and it turns the page's contract into "server props re-resolve on lifecycle transitions".
- **(P3)** EncounterSnapshot is a cross-aggregate projection (session + PC character rows: the pc arm carries hp/sp/attributes from pcDetailById) but its invalidation signal covers only one aggregate: PC vitals writes go through the pools actions → version-guard → publishCharacterPing on the character channel, while useEncounterSnapshot subscribes only to the encounter channel and compares only the encounter row's version. Under healthy realtime there is no polling, so a DM HP adjust or a player self-heal never triggers a watcher refetch until the next session event bumps the encounter version. The gap is latent today only because no watch component renders snapshot PC vitals (grep confirms: zone tokens show name/portrait, the enemies rail shows enemy HP which is session data) — the snapshot ships PC hp/sp/attributes that nothing displays. The DM console solved the same problem explicitly with per-PC character-channel listeners (use-combat-console.ts onPcPing, docs/realtime/ADR.md subscription matrix), so the asymmetry is known machinery, just absent from the watch path.
  - *DX impact:* The obvious next watch ticket — a party HP bar, exactly what spectators at a table want — will render data that is silently stale precisely when the realtime layer is working, with the 1.5s poll fallback masking the bug in dev. Nothing in the EncounterSnapshot type or the hook warns the implementer.
  - *Recommendation:* Cheapest now: document the constraint on EncounterSnapshot's pc arm ("vitals are as-of last session event under realtime; do not render without adding PC-channel invalidation") and/or drop the unrendered pc hp/sp/attributes from the projection until a consumer exists (minimal-footprint preference). When a consumer lands, either mount per-PC RealtimeChannelListeners in the watch view like the console, or publish an encounter ping from the pools actions when the character is a live combatant.
- **(P3)** The player write path is structurally non-optimistic: use-own-combat-event.ts documents that the client holds only the redacted snapshot, not the CombatSession the reducer needs, so a player's condition toggle waits for round-trip + ping-triggered refetch (sub-second realtime, ≤1.5s poll) with only an opacity-dim pending state (player-combat-state-control.tsx). This is a deliberate, documented trade-off — but it is also a ceiling: every future player-interactive feature (own token movement is the natural one, given UNN-315 exists for the DM) inherits the same latency or forces a snapshot-level optimistic reducer that doesn't exist.
  - *DX impact:* Each new player-side interaction ships with visible lag relative to the DM console's instant optimistic UI, and the gap will read as a bug to users. If/when someone adds a snapshot-level optimistic apply, it risks becoming a second, divergent reduce path unless designed against the session reducer.
  - *Recommendation:* Accept for now (correct call for conditions toggles). When a latency-sensitive player interaction lands, prefer a tiny overlay-only optimistic apply on the snapshot combatant (the player-overlay event set is small and closed — isPlayerOverlayEvent already defines it) rather than shipping the full session to players; note this direction in the watch view's module doc so the next implementer doesn't invent a third model.
- **(P3)** The startCombat two-write non-atomicity (session save then status flip, each version-guarded) is documented in events.ts with the claim 'The DM client contract (UNN-335) owns that reload-and-reissue path' — but no client implements it: useEncounterSetup's dispatch error arm only toasts ('This encounter changed elsewhere. Reload and try again.', error-message.ts), leaving the human as the retry loop and the optimistic state reverted with a now-stale versionRef. The recovery does work manually (re-applying startCombat is a reducer no-op), but no surface self-heals on a stale error — none call router.refresh() on failure, so after any stale rejection the page sits on outdated state until a manual reload.
  - *DX impact:* A one-line gap that turns every legitimate version conflict (second DM tab, player overlay write racing a DM edit) into a dead-end toast; as more writers join the session (own-events already landed), stale rejections get more common and each one costs a full manual reload.
  - *Recommendation:* Add router.refresh() on the 'stale' error arm of the shared dispatch (one place, if finding 1's extraction happens) so the page re-syncs and the user just re-taps; then fix the events.ts JSDoc to describe what the client actually does.
- **(P3)** addSetupCombatantsAction is the one session write that does not publish an encounter ping: events.ts, own-events.ts, and end.ts all call publishEncounterPing after a guarded save; setup.ts only calls revalidateEncounter. Consequence-free today (draft watchers see WaitingState; setup deliberately has no realtime subscription), but it breaks the otherwise-uniform invariant 'every encounter session write pings the encounter channel'.
  - *DX impact:* The next consumer that assumes the invariant — a second-device setup story, a campaign-page roster preview, or simply a watch view that starts showing the draft roster — will silently miss catalog-add updates, and the bug will be hard to spot because every other write path behaves.
  - *Recommendation:* Add the publishEncounterPing call to addSetupCombatantsAction (two lines, mirrors its peers) or, if deliberately omitted, say so in the JSDoc where the action otherwise advertises that its 'flow mirrors applyCombatEvent'.
- **(P3)** Enemy-queue localStorage entries are cleared only on explicit commit or cancel (enemy-catalog-browser.tsx returnToSetup/commit); an encounter that goes live, ends, or is deleted while a queue exists leaves an orphan unnamed-system:encounter-enemy-queue:{uuid} key forever (the non-draft redirect in app/combat/[shortId]/enemies/page.tsx never reaches the clear). The queue is also keyed by the internal encounter UUID rather than shortId — harmless in the DM's own browser, but the only place in the combat surfaces where the internal id crosses to the client persistence layer.
  - *DX impact:* Pure hygiene today (a few stale keys). Mild compounding risk as the pattern is reused — UNN-299 custom enemies is a plausible second staging queue — and a future 'restore your queue' affordance built on these keys would resurrect rosters for dead encounters.
  - *Recommendation:* Lowest-cost: clear the key in the non-draft redirect path or when the setup shell mounts for a non-draft encounter. Not worth more than a few lines; fine to fold into the UNN-299 ticket.
- **(P3)** Adding one per-combatant property fans out across 6–8 thin layers with no written recipe: foundation combatantSchema (+ Zod default for old persisted sessions), makeCombatant in session-factory.ts, a CombatEvent arm + reduce/* slice, the DM view-models (CombatantDetail / roster rows), PlayerCombatantBase + projectCombatant in player-snapshot.ts (where the redaction decision must be made consciously — the structural-absence design means forgetting this step omits the field from players, the safe default), a drawer section, and possibly ConditionsControls/watch rendering. The recently-landed counters feature traces this exact path (countersSchema in combatantSchema → reduce/counters.ts → counters in PlayerCombatantBase with an explicit visibility JSDoc → combatant-counters-section.tsx), proving the path is paved but memorized, not documented.
  - *DX impact:* Each property addition is a re-derivation exercise: which projections must it appear in, which may it, and what does each omission mean? The structural redaction makes mistakes fail safe (hidden from players) but also fail silent — a property the watch view should show just won't, with no error. For a solo developer returning after a gap, this is the highest-recall-cost operation in the subsystem.
  - *Recommendation:* Write a ten-line 'adding a per-combatant property' checklist (the counters commit is the worked example) in docs/initiative-tracker/ or the session.ts module doc, with one line forcing the visibility decision: 'decide explicitly whether the player snapshot carries it, and document why on PlayerCombatantBase'. No code change needed — the layering itself is right.

### Engine seam from the web side

The seam as built is one of the strongest pieces of architecture in this app, and — unusually — the practice is better than the paperwork. The composition root (apps/web/lib/game-engine.ts) is 37 lines: bind gameData and a default id generator once via createGameEngine, destructure 18 pre-bound boundary functions, done. Consumption discipline is near-perfect by inspection: 31 files import the bound functions from @/lib/game-engine; all other @workspace/game imports in app source are layer-barrel imports of types, vocabulary, deps-free pure helpers (canAfford, hydrateFormula, getMechanic, formatSignedBonus), or direct catalog display-lookups (getArchetype, archetypeDisplayName, getAilment) from the data barrel; there are zero deep-module imports and zero ad hoc gameData bindings outside one app integration test. The curried deps-first signatures impose no noise at app call sites — components call bound functions like ordinary functions — and the currying pays for itself in e2e fixtures, which bind deterministic id generators. The real weaknesses are structural-but-latent rather than active: the entire seam is convention with no lint enforcement (the engine barrel re-exports the unbound curried functions under identical names, and the package's wildcard exports permit the deep imports the engine-reorg README claims are blocked); the composition root's JSDoc claims catalog dependence is 'confined here' while ~28 files legitimately read the data barrel directly; and the module-scope bind makes the client bundle all-or-nothing over the full catalog. One genuine 'game logic in UI' instance surfaced (enemy heal-cap clamping in the DM drawer). None of these taxes the next ticket today; the enforcement gap is the one that compounds if left unaddressed, because the seam's current cleanliness is exactly what a lint rule could lock in cheaply right now while violations number zero.

**Strengths:**

- The composition root delivers exactly what it promises at call sites: bound functions read as plain domain calls — `resolveTalentsForSheet(character.gainedTalents, character.activeArchetypeKey)` in explore/talents.tsx, `reduceCharacter(current, edit)` inside useOptimistic in use-character.tsx — with zero currying noise, no threaded gameData, no newId plumbing. 31 files consume `@/lib/game-engine` and none of them betray that the underlying functions are curried.
- Exactly one production binding site, verified: grep for `createGameEngine` and `gameData` across apps/web finds only lib/game-engine.ts in app source (the single exception is one app integration test). The double-call form `createCombatSession(() => id)(setups)` appears only in tests and e2e fixtures, where binding a deterministic id generator is precisely the point — the curried design is earning its keep there, not just imposing ceremony.
- Zero deep-module imports: every one of the ~180 `@workspace/game` import lines in apps/web goes through the three layer barrels, despite the package.json wildcards permitting deeper reaches. The classification is clean — foundation for types/vocab/Result, engine for types + genuinely deps-free helpers and constants (canAfford, hydrateFormula, getMechanic, MASTERY_RANK, VICTORIES_PER_LEVEL), data for display lookups, composition root for everything that touches GameData.
- use-character.tsx is a model seam consumer: the optimistic client reduce and the server's persistence path run the same bound reduceCharacter, so 'the same derivation the server runs' is structurally true rather than aspirational — and the provider centralizes it so no component ever touches the reducer directly.
- The 'no game logic in the UI' convention substantially holds: a sweep of Math.* usage in components found only display clamping (vital-bar percentage, input steppers) plus the one enemy-vitals exception reported below; per-tab data shaping (resolveTalentsForSheet, getArchetypeDisplay, buildLineageAtlas, resolveCatalogEnemyStatblocks) genuinely lives engine-side and tab roots call one helper as CLAUDE.md prescribes.
- The deps-first/deps-free split is principled, not accidental: ports.ts documents why mechanics dispatch (getMechanic) is engine-owned behavior rather than a data port, and the helpers components import raw (isInheritableSkill, unmetPrerequisites, toCombatantSetup) all take already-resolved domain values — checked signatures confirm none of them secretly need a catalog.

**Findings:**

- **(P2)** The seam is convention-only — nothing enforces it. The engine barrel (packages/game/src/engine/index.ts) wholesale re-exports the unbound curried boundary functions ('`export * from "./create-engine"`', './character/reduce-character', './encounter/session-factory', all reduce/* slices) under the exact same names the composition root re-exports bound, so `reduceCharacter` resolves from both `@workspace/game/engine` (curried, takes a GameData slice) and `@/lib/game-engine` (bound). The package.json wildcard exports (`"./engine/*": "./src/engine/*.ts"`) additionally let app code deep-import any engine internal — contradicting docs/engine-reorg/README.md's claim that 'The package exports map already blocks reaching internals from apps/web'. The ESLint boundary rule is acknowledged as still TODO in that README (step 7), and no rule for the app→raw-curried direction was ever planned; packages/eslint-config has no no-restricted-imports/boundaries rules at all. Today there are zero violations in app source (verified by grepping every @workspace/game import), so the discipline holds purely by convention plus TypeScript arity errors.
  - *DX impact:* Every contributor (including future Claude instances) must infer an unwritten rule — 'GameData-slice boundary functions come from @/lib/game-engine; deps-free helpers, types, and constants come from the layer barrels' — that is encoded only in one JSDoc in apps/web/lib/game-engine.ts. Editor auto-import offers both symbols for the 18 bound names; picking the barrel one yields a confusing slice-type error (or, for slice-last helpers like resolveTalents, typechecks fine once you import gameData, silently eroding the composition root). The cost is small per incident but recurs on every feature that touches the engine, and the first violation that lands sets a precedent grep can no longer distinguish from the sanctioned pattern.
  - *Recommendation:* Add one flat-config ESLint no-restricted-imports rule scoped to apps/web (excluding lib/game-engine.ts and test/e2e files): forbid importing `createGameEngine`, `gameData`, and the 18 bound names from `@workspace/game/engine`/`@workspace/game/data`, and forbid any `@workspace/game/*/**` deep path. This is the lint half the engine-reorg README already planned but never landed, extended one step to cover the app side; an hour of work that converts the convention into a signal.
- **(P3)** The composition root's JSDoc overclaims its own coverage: apps/web/lib/game-engine.ts says 'the catalog/demo-flag dependence stays confined here', but 28 non-test app files import catalog getters from `@workspace/game/data` directly — ~20 components (getArchetype in mechanic-widget.tsx and atlas/archetype-detail-panel.tsx; archetypeDisplayName in sheet-header.tsx, character-card.tsx, import-pcs-panel.tsx and 3 more; getAilment/AILMENTS in conditions-controls.tsx, ailment-editor.tsx, ailment-list.tsx, watch-enemies-rail.tsx; WEAPONS/ARMOR/CONSUMABLES/ACCESSORIES in add-item-dialog.tsx; getEnemy/getEnemyFamily in enemy-catalog-browser.tsx; getSkill, getTalent, getSideEffect, INITIATE_ARCHETYPES elsewhere) plus lib files (labels.ts, two action schemas, four db writes, load-encounter-snapshot.ts). The design is actually coherent — the demo-archetype flag is resolved inside the data registry (packages/game/src/data/archetypes/registry.ts merges DEMO_ARCHETYPES at module scope), so direct getters and the bound engine can never disagree — but the documented rule and the practiced rule differ, and the GameData port's swap-ability does not extend to these ~28 scattered read sites.
  - *DX impact:* A developer reading game-engine.ts's JSDoc concludes direct data imports are violations and either 'fixes' correct code or threads lookups needlessly; a developer reading the codebase concludes the JSDoc is aspirational and stops trusting it. Concretely low day-to-day cost (lookups are sync, demo-flag-safe, and the MVP rules out a DB-backed catalog), but it is the kind of doc/code drift that misleads exactly the audience this project optimizes for — future Claude instances taking CLAUDE.md and JSDocs as authoritative.
  - *Recommendation:* Two-line JSDoc fix in apps/web/lib/game-engine.ts: state the real two-tier rule — GameData-slice boundary functions are bound here; deps-free engine helpers and direct catalog display-lookups (`getArchetype`, `archetypeDisplayName`, `getAilment`, …) import their layer barrel directly, and the demo flag is honored inside the data registry so both surfaces agree. Add the same sentence to CLAUDE.md Code Conventions. Do NOT build a catalog facade — 28 stable read sites for hardcoded data don't justify one.
- **(P3)** The seam has two parameter conventions for data-dependent functions, and the second one has no app-side story. Boundary functions are deps-first curried and bound in createGameEngine; but the engine barrel also exports slice-LAST, non-curried helpers — e.g. `resolveTalents(gainedTalents, activeArchetypeKey, lookups: Pick<GameData, "getArchetype" | "getTalent">)` in packages/game/src/engine/character/talents/utils.ts — with no bound counterpart. The one app consumer (`apps/web/lib/__tests__/character-sheet.test.ts`) resolves this by importing `gameData` from `@workspace/game/data` and threading it by hand — the exact pattern the composition root exists to prevent, in the app tree.
  - *DX impact:* When a feature needs one of these helpers in app code, the path of least resistance is copying the test's pattern: import gameData, thread it inline. Each occurrence is locally harmless and globally corrosive — the 'one production binding site' invariant decays one helper at a time, and grep can no longer answer 'where is the catalog bound?'. Today the cost is zero (test-only); the finding is about the missing rule for the first real occurrence.
  - *Recommendation:* Add one sentence to game-engine.ts's JSDoc: 'If app code needs a slice-taking engine helper that isn't bound here, add it to createGameEngine (one line in the sweep) rather than importing gameData.' Optionally move the test's resolveTalents call behind a bound export the same way. No refactor of the engine's internal slice-last helpers is warranted — that's engine-internal style, out of the app's scope.
- **(P3)** The composition root is all-or-nothing for the client bundle: `createGameEngine(gameData)` executes at module scope of apps/web/lib/game-engine.ts, and gameData closes over every catalog (ARCHETYPES, ENEMIES, all skills/items/talents — packages/game/src/data/game-data.ts). Any client component importing a single bound function — e.g. use-character.tsx ('use client') importing reduceCharacter for the optimistic sheet — therefore pulls the entire data layer plus all 18 boundary modules into that route's bundle. Tree-shaking and optimizePackageImports (configured in next.config.ts) cannot split a runtime object destructure. The character sheet thus ships the full 5e enemy catalog it never renders.
  - *DX impact:* Today this is cheap — the whole data layer is ~2.6k lines of source — and game data is public, so the only cost is bytes. But the cost grows linearly with the enemy/skill catalogs (the area most likely to grow as DM tooling expands), it is invisible (no one will notice the sheet bundle absorbing each new enemy), and the single-object design means there is no incremental way to opt a surface out without restructuring the root.
  - *Recommendation:* Accept it now; record it as a known property in the game-engine.ts JSDoc so the eventual bundle-analysis ticket has a starting point. If it ever matters, the fix is mechanical because createGameEngine is a uniform sweep: split the bind into two composition modules (character-engine vs encounter/enemy-engine) sharing one gameData import — no engine changes needed. Do not pre-build that split today.
- **(P2)** One concrete instance of a game rule living in the UI layer: the DM drawer's enemy-vitals control computes the post-adjustment absolute HP in the component — heal is `Math.min(hp.max, hp.current + amount)` and max-HP lower is `Math.max(0, hp.max - amount)` in apps/web/components/combat/combatant-vitals-section.tsx (lines ~206, ~222) — and dispatches an absolute-set `adjustEnemyVitals` event. The engine reducer (packages/game/src/engine/encounter/reduce/enemy-vitals.ts) floors the incoming value at 0 and clamps current under a lowered max, but does NOT cap current at max. So the 'heal cannot exceed max' rule exists only in this one component, while the 'HP floors at 0' rule exists only in the engine — split-brained clamping the component's own JSDoc documents ('the reducer floors max at 0').
  - *DX impact:* Any second write surface for enemy vitals — the player watch view's own-combatant controls, a command-palette batch, a future automation — must remember to re-implement the max-cap or it can push an enemy's currentHP above max through a perfectly valid event. The reducer accepting over-max values means the session jsonb can persist states the UI believes impossible, which is exactly the class of bug the 'never put game logic in the UI layer' convention exists to prevent.
  - *Recommendation:* Move the cap into the adjustEnemyVitals reducer (clamp currentHP to [0, max] on every set) and delete the component-side Math.min — a few engine lines plus a unit test, and the Stryker suite will then guard the rule. Keep the UI computing absolutes if you like the popover ergonomics; the reducer just stops trusting them.

### DX walkthrough: paved path measurement

The paved path is real and most of its hard surface is excellent: the layers a developer should never re-derive (optimistic concurrency, stale-retry, cross-tab broadcast, realtime invalidation, debounce/blur races, cache revalidation) are each solved exactly once at a choke point — bumpCharacterVersionGuarded carries the realtime ping so step "realtime-sync" is zero files, the select-all loader plus `HydratedCharacter = CharacterRow & {…}` makes the read path zero files, and the compile-enforced EDIT_SURFACE_CLASS map makes misclassifying a write surface impossible rather than discouraged. A battle-cry implementation is 11 mostly-mechanical touches with no genuinely hard step, which is a strong result. The friction is concentrated in the middle tier: each scalar field still costs a hand-cloned action + schema + DB-wrapper triple (~120 lines) plus a Result-reshape in the component, even though the codebase already contains the collapsed shapes (narrative's field discriminator, identity.ts's unexported runIdentityUpdate) that would make the 10th field cheaper than the 1st; and the README that every new write ticket is directed to start from has drifted in its import paths, its mechanics section, and its hand-mirrored concurrency table, while simultaneously mandating an aggregate-folder layout that zero character actions actually follow. None of this blocks the next 25 tickets — but the boilerplate triple and the doc drift will tax each of them a little, and both are cheap to fix with material that already exists in the repo.

**Strengths:**

- The version-guard choke point is the standout: bumpCharacterVersionGuarded (lib/db/writes/version-guard.ts) bundles the conditional UPDATE, atomic version bump, stale-vs-deleted disambiguation, AND the Ably realtime ping (publishCharacterPing at line 102) into the one primitive every write composes through — so a new field gets optimistic concurrency, cross-user realtime sync, and cross-tab broadcast with literally zero realtime wiring.
